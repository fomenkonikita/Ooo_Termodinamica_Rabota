## Листы таблицы
- **Сотрудники**: A=TG ID, B=Имя, C=Тип (объект/водитель/сервис), D=Локация(текущая), E=Активен, F=Начало работы, G=Конец работы, H=Рабочие дни
- **Локации**: A=Название, B=Широта, C=Долгота, D=Радиус(м)
- **Журнал**: A=Дата, B=Имя, C=Тип, D=Объект, E=Приход, F=Уход, G=Отработано, H=Статус, I=Посл.активность, J=TG ID (с 30.06.2026, новые записи)
- **Точки водителей**: A=Дата, B=Имя, C=Время, D=Широта, E=Долгота
- **Месячный лист** (напр. "Июль 2026"): A=Имя, B..эта дней=дни, +1=Итого. Выходные серые, текущая смена — зелёная.
- **GPS лог**: A=Дата, B=Время, C=TG ID, D=Имя, E=Объект, F=Lat, G=Lon, H=Accuracy, I=Подозрение
- **Уведомления**: A=Дата, B=Время, C=Сотрудник, D=Тип, E=Запланировано, F=Статус, G=Текст
- **Дашборд**: сводный лист для директора (см. ниже)

## Типы сотрудников
- **объект** — приход/уход с проверкой GPS-радиуса от объекта (кнопки "На объекте")
- **водитель** / **сервис** — начало/конец смены + точки маршрута, без проверки радиуса (визуально разные подписи, поведение идентично)

## Расписание джоб (APScheduler)
- `job_schedule_check` (5 мин) — напоминания сотруднику до начала/конца смены, сводка админу
- `job_resync_green` (5 мин) — переподсветка зелёным тех, кто на смене (самопочинка)
- `job_update_dashboard` (5 мин) — синхронизация переименований (`sync_employee_names`) + полная пересборка дашборда
- `job_close_21` (21:00) — авто-закрытие всех открытых смен (макс 8ч) + кнопка "Я ещё на работе"
- `job_remind_2350` (23:50) — напоминание продлившим смену
- `job_hard_close` (23:55) — жёсткое закрытие всех оставшихся
- `job_reconcile` (00:10) — сверка пропусков (Журнал vs месяц) + проверка целостности часов (`verify_journal_integrity`)

## Хронология фиксов — 30.06.2026

### Критичные баги с потерей данных
1. **`auto_close_entry` писал текст "⚠️ авто" вместо часов** в колонку часов → SUM() игнорировал авто-закрытые смены. Исправлено: пишутся реальные часы.
2. **Часы за день перезаписывались, а не суммировались** (`_write_monthly`/`_mark_monthly_auto`) — если у сотрудника несколько смен за день (обед, повторный приход), в табеле оставалась только последняя смена. Исправлено: обе функции пересчитывают сумму всех закрытых смен дня из Журнала.
3. **Одна историческая запись с неверными часами** (29.06, Никита, было 0:00 вместо 2:00) — причину в коде установить не удалось (история git пересоздана), но добавлена защита: `verify_journal_integrity()` сверяет каждую запись Журнала с реальной разницей приход/уход, чинит и алертит, запускается ночью.

### Структурная причина повторяющихся "новый сотрудник не попал в табель"
- **Реальная причина (не баг логики!)**: транзитные SSL-ошибки сети между Render и Google Sheets API. Ни один вызов `.execute()` не имел retry — единичный сбой сети тихо терял запись.
- Исправлено: `_execute()` — обёртка с 3 попытками и backoff на все вызовы `.execute()` в sheets.py.

### Дубль регистрации / краш-петли
- **Дубль сотрудника**: бот принял кнопку "🚪 Ушёл" за имя нового сотрудника (баг с устаревшим состоянием + порядком хендлеров telebot). Исправлено: `reg_name`/`reg_type` проверяют, не зарегистрирован ли уже пользователь.
- **Краш-петля от Telegram API**: необработанное исключение в одном хендлере (просроченный callback) рушило весь процесс, 4+ рестарта за 5 минут. Исправлено: глобальный `exception_handler` в TeleBot + `safe_answer_callback`/`safe_clear_markup` обёртки.
- **Конфликт двух инстансов**: забытый локальный `python bot.py` дрался с Render за getUpdates от Telegram, объясняя часть "случайных" сбоев. Процессы остановлены, зафиксировано правило: бот живёт только на Render.

### Переименование сотрудников ломало статистику
- Журнал/лист месяца хранят строки по ИМЕНИ, а имя в Сотрудники — редактируемое поле. Переименование (Роман → Роман Рященко) осиротило все его старые записи.
- Исправлено структурно: `record_arrival` пишет TG ID в Журнал!J (стабильный ключ). `sync_employee_names()` сверяет по TG ID и автоматически переименовывает дрейфующие записи. Запускается каждые 5 мин.

### Дашборд для директора
- Новый лист "Дашборд": живой статус "кто сейчас на работе", сводка на сегодня, GPS-аномалии, месячная сводка по сотрудникам, реестр уведомлений (план/факт/пропущено), оформление (группировка пустых строк, рамки, бандинг, KPI-плашки).
- **Race condition при параллельных пересборках** (рваные данные) → добавлена блокировка.
- **Блокировка по ошибке отбрасывала события вместо очереди** (уход+приход подряд → второе событие терялось) → переделано на coalescing-паттерн (dirty flag, повтор после текущей пересборки).
- Автообновление при КАЖДОМ событии Журнала (приход/уход/авто-закрытие), не только по таймеру 5 мин.

### Защита от подмены GPS
- Telegram не даёт флага "это мок-локация", только `latitude/longitude/horizontal_accuracy`.
- Эвристика (не блокирует, только алерт админу): нет accuracy ИЛИ координаты побитово совпадают с прошлым визитом на тот же объект (реальный GPS всегда даёт небольшой разброс).
- Лог всех замеров — лист "GPS лог".

### Скорость и UX
- Отметка прихода/ухода раньше делала 8-10 последовательных запросов к Sheets API подряд, прежде чем ответить пользователю. Исправлено: критичная запись (Журнал) — мгновенно, остальное (месячный лист, локация, GPS-проверка) — в фоновом потоке.
- Защита от "фантомных" смен: если сотрудник жмёт "Пришёл" после уже состоявшегося сегодня цикла приход+уход — переспрашивается подтверждение.
- Реестр уведомлений ("📋 Уведомления", только админ) — что запланировано, что отправлено, что пропущено.
- Privacy Policy страница (`/privacy` на сервере бота) — для требования BotFather.

## Известные ограничения / технический долг
- Журнал-записи **до 30.06.2026 вечера** не имеют TG ID в колонке J — для них автосинхронизация переименований не сработает (разовый хвост, не растущая проблема).
- Render free-tier — сеть иногда нестабильна (SSL-ошибки), retry помогает, но не гарантирует 100%. При продолжении нестабильности — переход на платный план.
- График на дашборде (диаграмма часов по сотрудникам) — не сделан, можно добавить через `batchUpdate addChart`.
- MAX-мессенджер (параллельно с Telegram) — отложено: создание бота требует верификации юрлица через Госуслуги (ИП/ООО), пользователь физлицо это сделать не может напрямую.
