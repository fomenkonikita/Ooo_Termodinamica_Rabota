## Архитектура — НЕЛЬЗЯ МЕНЯТЬ без причины

### Структура процесса (критично)
```
Flask (daemon thread)       ← health check /  → всегда 200
Watchdog (daemon thread)    ← os._exit(1) если нет API вызовов 30+ мин
Scheduler (background)      ← APScheduler jobs
bot.infinity_polling()      ← MAIN THREAD — так задумано, не менять
```

**Почему Flask daemon, polling на main:** если Flask на main — бот работает даже когда polling упал,
но OOM убивает процесс за 60 сек (512MB Render free limit). Оригинальная архитектура = polling на main,
Flask daemon. Это работало до 01.07.2026 и после возврата к этой схеме работает снова.

**Инцидент 01.07.2026:** перенёс Flask на main, polling в daemon — это вызвало OOM каждые 60 сек.
Причина: пик памяти при старте (все модули + Flask + job_update_dashboard при старте).
**Никогда не запускать job_update_dashboard при старте процесса — только через scheduler.**

### Scheduler jobs (каждые N минут)
- `job_schedule_check` — каждые 5 мин: напоминания по графику, сводка админу
- `job_resync_green` — каждые 5 мин: зелёная подсветка в листе месяца
- `job_update_dashboard` — каждые 5 мин: закрыть orphaned + sync names + resync totals + rebuild dashboard
- `job_close_21` — 21:00: авто-закрытие всех открытых смен + кнопка "ещё на работе"
- `job_remind_2350` — 23:50: напоминание продлившим смену
- `job_hard_close` — 23:55: жёсткое закрытие оставшихся
- `job_reconcile` — 00:10: сверка вчерашнего дня, закрытие orphaned, целостность Журнала
- `job_keepalive` — каждые 10 мин: пинг себя чтобы Render не засыпал

## Google Sheets — структура листов

### Журнал (источник правды)
| A | B | C | D | E | F | G | H | I | J |
|---|---|---|---|---|---|---|---|---|---|
| Дата | Имя | Тип | Объект | Приход | Уход | Отработано | Статус | Посл.активность | TG ID |

Статусы H: `✅` норм, `⚠️ авто` авто-закрытие, `⏳ продлено до 23:55` продлил

**Дата хранится как НАСТОЯЩАЯ дата** (не текст), хотя выглядит как "01.07.2026".
Это важно для формул — сравнивать с TODAY(), не с TEXT(TODAY();"дд.мм.гггг").

## Известные баги — исправлены, понимать почему

### OOM на Render free tier (01.07.2026)
Flask на main thread + job_update_dashboard при старте = пик >512MB = OOM каждые 60 сек.
**Решение:** Flask в daemon, polling на main, NO startup jobs. Только scheduler.

### Orphaned записи (когда bot падает в 21:00)
Если бот падает во время job_close_21 — записи остаются открытыми навсегда.
Сотрудник не может войти (бот думает что смена открыта).
Дашборд показывает их как "на работе" (неверно).
**Решение:** `close_orphaned_entries()` в `job_update_dashboard` (каждые 5 мин).

### Дашборд показывал записи прошлых дней (01.07.2026)
`get_open_entries_all()` не фильтрует по дате — возвращает ВСЕ открытые записи.
Дашборд показывал вчерашних незакрытых как "сейчас на работе".
**Решение:** в `_rebuild_dashboard` фильтр `date == today_str` перед записью в блок 1.

### Httplib2 зависание навсегда (30.06.2026)
`httplib2.Http()` без timeout — при тишине от Google висит бесконечно.
Вся система зависала на 45+ минут без единой строки в логах.
**Решение:** `AuthorizedHttp(creds, http=httplib2.Http(timeout=30))` в `_svc()`.

### Двойная запись часов при нескольких сменах в день (30.06.2026)
`_write_monthly` перезаписывала ячейку часами последней смены, стирая предыдущие.
**Решение:** суммируем ВСЕ закрытые смены за день из Журнала.

### Сотрудник не попадает в месячный лист (30.06.2026)
SSL-ошибки тихо глотались — запись в Журнал проходила, но все остальные вызовы падали.
**Решение:** `_execute()` обёртка с 3 попытками и backoff на все вызовы Google API.

### Переименование сотрудника ломает статистику (30.06.2026)
Имя — нестабильный ключ. При переименовании в Сотрудники старые записи "осиротели".
**Решение:** `sync_employee_names()` сверяет по TG ID (колонка J в Журнале, стабильный ключ).
